# 2026-04-29 1937 PH5 vs KCM teamstats gid typo — 6 row gid corrections

**Date:** 2026-04-29
**Scope:** 4 source CSVs across `data/`
**Updated:** 6 row gid values (PH5194704130 → PH5193704130)
**DB rows updated:** 2 (in `retro.teamstats`)

## Background

Layer 1 verification (post-Phase-2-real-ingest, 2026-04-29) check L1.6 (games ↔
teamstats symmetry, `scratch/l1_06_games_teamstats_symmetry.sql`) surfaced:
- 1 gid in `retro.games` with no matching `retro.teamstats` rows: `PH5193704130`
  (1937-04-13 KCM @ PH5, exhibition, played at HOU06, final score KCM 4 – PH5 5).
- 1 gid in `retro.teamstats` with no matching `retro.games` row: `PH5194704130`,
  whose 2 rows (one per team: KCM and PH5) carried correct `date='1937-04-13'`,
  correct team codes, and correct scores matching the canonical 1937 game.

The 12-char gid `PH5194704130` decodes as team prefix `PH5` + date `19470413` +
game number `0` — but the row's own date column says 1937-04-13. Single-digit
year typo in the gid string: `1937` → `1947` (3→4).

This is NOT the staging-duplicate pattern of CLAUDE.md §4a Item 23 — there are
no canonical clean copies elsewhere in the source CSVs. The 2 mistagged rows
are the only teamstats representation of this game.

## Why dry-run did not catch it

The typo is a valid VARCHAR(12) string and a valid CSV value at the column
level. `phase2_pk_uniqueness_scan.py` did not flag it because it doesn't
violate any PK constraint (in fact, gid `PH5194704130` is unique in
teamstats by accident — there's no real 1947 game to collide with, and
teamstats has no DB-enforced PK). The mismatch only surfaces as a
cross-table assertion: every `games.gid` should appear in `teamstats`, and
vice-versa. That structural check is Layer 1's L1.6, not a value-level
transform.

## Scope of the typo in source data

The same typo gid appeared in 4 distinct source CSVs (Retrosheet ships
multiple aggregations of the same data; only the first is read by ingest):

| File | Lines | Occurrences | Read by ingest? |
|---|---|---|---|
| `data/1937/1937teamstats.csv` | 18, 19 | 2 (KCM + PH5 rows) | Yes |
| `data/teamstats.csv` | 123224, 123225 | 2 (KCM + PH5 rows) | No (cross-year aggregate) |
| `data/teamlogs/1937KCM_stats.csv` | 3 | 1 (KCM row) | No (per-team alternate) |
| `data/teamlogs/1937PH5_stats.csv` | 3 | 1 (PH5 row) | No (per-team alternate) |

Total: **6 occurrences across 4 files**.

The canonical correct gid `PH5193704130` was already present in the
corresponding gameinfo / per-team log files
(`1937gameinfo.csv`, `gameinfo.csv`, `1937KCM_log.csv`, `1937PH5_log.csv`),
so the typo was isolated to teamstats-shape files — not propagated.

## Resolution

### Source-data fix

Replaced `PH5194704130,` → `PH5193704130,` in all 4 files (replace_all on
the unique gid prefix; no other occurrences of either string in any file).
Each file's `PH5194704130` count dropped to 0; canonical `PH5193704130`
count rose by the matching delta.

### DB fix

Inside a single transaction:

```sql
UPDATE retro.teamstats
   SET gid = 'PH5193704130'
 WHERE gid = 'PH5194704130'
   AND date = '1937-04-13'
   AND team IN ('KCM','PH5')
   AND stattype = 'value';
-- UPDATE 2
COMMIT;
```

Paranoid WHERE clause (gid + date + team + stattype) provides defense in
depth; gid alone matched the same 2 rows since `PH5194704130` was unique in
the corpus.

## How to reverse

To restore the source-data typo (e.g. for re-ingest verification against
historical state): in each of the 4 files, replace
`PH5193704130,` → `PH5194704130,` only on the matching team-suffix lines:

- `data/1937/1937teamstats.csv` lines 18 (KCM row) and 19 (PH5 row)
- `data/teamstats.csv` lines 123224 (KCM) and 123225 (PH5)
- `data/teamlogs/1937KCM_stats.csv` line 3
- `data/teamlogs/1937PH5_stats.csv` line 3

Note: adjacent lines in these files contain canonical gids for nearby games
(e.g. `PH5193704120` for the 1937-04-12 game on lines 16-17 of
`1937teamstats.csv`). When reverting, target only the lines whose date
column reads `19370413`, not adjacent dates.

To revert the DB:

```sql
UPDATE retro.teamstats
   SET gid = 'PH5194704130'
 WHERE gid = 'PH5193704130'
   AND date = '1937-04-13'
   AND team IN ('KCM','PH5')
   AND stattype = 'value';
```
