### Diagnosis before fix

**Project context:** 2026-05-03, Phase 3B closure session, items 3 and 4. Both items had carried "prompt-calibration items" framing in the backlog since Phase 3A Step 3 (2026-04-30). The closure session began by surfacing diagnostic information on each before proposing any fix — current state of the relevant rule, provenance of the original observation, characterization of the deviation.

**Observation:** Item 3 (era-disambiguation team rule partially ignored) was diagnosed first. The diagnostic revealed the evidence was stale — three days old, no verbatim artifact preserved, no eval-framework reproduction possible (no Washington-1920s question in `benchmarks.py`), prompt had changed substantially since the observation. The "fix" became "close to watch list, no edit." Item 4 (narrative + 2-match disambiguation calibration) got the same three-piece diagnostic and surfaced a completely different shape: the original observation was equally stale, but the diagnostic uncovered an active rule conflict in the current prompt. The narrative-disambiguation bullet said `ask_user`; the Dynamism section added later (item 7's resolution) said don't bounce back, answer all interpretations. The two rules contradicted on Ken-Griffey-shaped questions, and the agent's original "deviation" was the agent already preferring the Dynamism rule over the disambiguation rule. The fix wasn't calibration — it was rule reconciliation.

**Principle:** The discipline of diagnosing before fixing pays off in multiple distinct ways, not just one. Sometimes diagnosis reveals there's no problem worth fixing — evidence has aged out, the surrounding code has changed, or the observation was a one-off that doesn't reproduce. Sometimes diagnosis reveals the presumed problem is the wrong frame entirely — what looked like a calibration miss is actually a rule conflict, what looked like a regression is a stochastic variant, what looked like a one-off bug is a systemic gap. Skipping diagnosis and jumping to fix means the calibration miss gets "tightened" without the rule conflict being noticed (and the conflict re-surfaces next session), or the stale evidence drives an unnecessary prompt edit (and the prompt grows for no reason). Diagnosis time is leverage; the time spent surfacing context is cheaper than the time spent untangling a misframed fix.

**PM application:** When AI-system bugs are reported to a PM, the temptation is to route them to engineering with a "please fix" framing. The discipline that pays off is asking "what would diagnosis reveal here?" before scoping the fix — even thirty minutes of structured diagnostic work (current state of the relevant code or prompt, provenance of the original observation, characterization of the deviation) frequently changes the action. Some reported bugs are stale and don't reproduce on current state; some are misframed and reveal a different underlying issue when surfaced carefully; some are real but smaller in scope than first reported. The PM's job is to insist on this diagnostic step before greenlighting an engineering effort — not because engineering doesn't know to diagnose, but because the bug-report → fix-scope handoff often loses the framing space. In an AI-system context where rules and prompts iterate frequently, this matters more than in static-code work: the system you're fixing today is not the system the bug was reported against.

### Rule conflicts compound silently

**Project context:** 2026-05-03, Phase 3B closure, item 4 reconciliation. Discovered through item 4's diagnostic surfacing.

**Observation:** The narrative-disambiguation rule was added during Phase 3A Step 3 (2026-04-30) as part of the original disambiguation tiered logic — narrative-shape questions with multiple matches should escalate to `ask_user`. The Dynamism section was added during Phase 3B Step 2 (2026-05-02) as part of item 7's resolution, responding to a different observation about multi-interpretation handling — for genuinely ambiguous questions, answer multiple interpretations at once and don't bounce the question back. Both rules were individually correct against the observations they were responding to. Neither was wrong when written. But they contradicted each other on a specific question shape (narrative + 2-3 plausible matches): disambiguation said `ask_user`, Dynamism said don't bounce back, answer all interpretations. The contradiction was invisible until item 4's diagnostic surfaced both rules in the same conversation. Phase 3B Step 2 added the Dynamism section without revisiting the disambiguation logic that was already in the prompt; the rules drifted into contradiction without either author (the same Claude collaborator across different sessions) noticing. The agent had been silently picking the Dynamism rule over the disambiguation rule for several days — reflected in the original Phase 3A Step 3 observation that flagged the "deviation" as actually-preferable UX.

**Principle:** When prompts iterate additively across multiple sessions, rule conflicts compound silently. Each added rule is reviewed against the local observation it's responding to, not against the existing rule landscape. Two rules that touch the same behavioral territory can drift into contradiction without either being individually wrong, and the contradiction isn't visible at any single review point — only at the combined moment when both rules fire on the same input. The cheap mitigation is a brief audit of nearby rules for consistency every time a new rule lands; the expensive mitigation is retrospective reconciliation after the conflict has already shaped agent behavior. Cross-references in the prompt itself ("same 'answer first, ask second' pattern as stat questions") make conflicts harder to introduce in the first place because the next author sees the dependency before adding new behavior.

**PM application:** AI products that ship behavioral rules — system prompts, policy guardrails, tone specifications, content filters — accumulate the same conflict-compounding risk. Each rule lands during a specific incident response or feature launch, gets reviewed against its local motivation, and is added to a growing set without consistency review across the set. The PM application is twofold: treat the prompt or rule set as a coherent specification with internal cross-references rather than as an append-only log, and require pairwise consistency checks when new rules touch existing behavioral territory. Practically, this means budgeting time during prompt iteration for "look at the rules already in this section" review, not just "look at the observation that motivated the change." Prompt-engineering velocity feels fast because each iteration is small, but the silent-conflict cost is paid in retrospective reconciliation work that's harder to schedule and harder to justify than incremental adds. Audit cycles on existing rules are an under-priced PM activity.

### Watch lists vs deferred backlogs

**Project context:** 2026-05-03, Phase 3B closure, item 20 (must_contain subject-name brittleness). One stochastic observation across 12 eval runs surfaced as a candidate framework change.

**Observation:** The Q7 fail in `eval/results/2026-05-03_111550/` surfaced a real framework brittleness — the agent answered Aaron's career HR total correctly but didn't restate the subject's name, and the `must_contain` check failed because it required `'aaron'` as a literal substring. The "obvious" fix was to soften the must_contain entries on Q1, Q5, Q6, Q7, Q8, Q9 to drop subject-name requirements; the factual-correctness pair (`expected_answer` + `sql_scalar_matches_answer`) already validated the substantive answer. But the right call was to track the observation as a watch item (item 20) with a clear reopen trigger ("if the pattern recurs across multiple runs OR surfaces on Q1/Q5/Q6/Q8/Q9"), rather than to fix it now. One observation across 12 runs was below the threshold for changing 6 spec entries preemptively. The "deliberately not acting now" prose in item 20 explicitly captured the reasoning so future-self wouldn't lose the context that justified the deferral.

**Principle:** Not every observed issue should be fixed when first observed. Single-observation framework changes have real costs — they remove signal that was working most of the time, they accumulate spec edits that have to be maintained, they confuse the relationship between framework and observation. The discipline that pays off is encoding framework changes based on observed patterns, not anticipated ones. A watch item with a named reopen trigger is structurally different from a deferred backlog item: the trigger makes the deferral testable, the prose captures the reasoning, and future-self has a clear signal for when to act. Deferred backlogs become pile-ups because there's no testable condition for when to engage; watch lists stay tractable because each item carries its own actionability criterion.

**PM application:** AI product feedback inevitably includes a long tail of low-frequency issues that are individually real but collectively dilute prioritization. The PM trap is treating each observation as backlog-eligible — every reported issue gets an item, items accumulate faster than they're closed, and prioritization becomes guesswork against an unbounded list. The discipline that pays off is distinguishing watch items (single observations with named reopen triggers) from backlog items (patterns confirmed across multiple observations or with clear product-impact framing). Each watch item should answer "what would have to be true for this to become urgent?" and that condition should be testable against the next round of feedback. This compresses the prioritization conversation: watch items don't compete for engineering time until their triggers fire, and backlog items have clear "promotion criteria" rather than vibes-based prioritization. The product lead who uses this framing has more clarity on what the team is actually working on and why.

### Examples in prompts are sticky

**Project context:** 2026-05-03, Phase 3B closure, item 4 reconciliation. The narrative-disambiguation rewrite included an illustrative multi-bio example. Initial draft → senior-advisor refinement.

**Observation:** Initial draft used "Ken Griffey Sr. — outfielder 1973-1991, three-time All-Star with the Big Red Machine; Ken Griffey Jr. — Hall of Fame outfielder 1989-2010, 630 career HRs." Refined to "Ken Griffey Sr. — outfielder, primarily Cincinnati Reds in the 1970s-80s; Ken Griffey Jr. — Hall of Fame outfielder, primarily Seattle Mariners in the 1990s-2000s." The original draft baked in specific factual claims (year ranges, three-time-All-Star count, 630 HR count, HOF status) that the agent could plausibly mimic in production responses if it treated the example as a template rather than a format. The refinement traded specifics (HR counts, All-Star counts, debut/retirement-year precision) for broader patterns (position + primary team + decade range) that are easier to verify against the database than stat-line claims. The upstream "Trusting the data" section was already authoritative on actual fact-sourcing, but the example-refinement was defense-in-depth: if the agent over-mimics, the failure mode is gentler. This wasn't a hypothetical concern — the same prompt has another section, the stat-question disambiguation bullet, using "Ken Griffey Sr. hit 14; Ken Griffey Jr. hit 22" as its example. Those numbers are illustrative, not real (Sr.'s 1990 HR total was 3, not 14). Examples can model the wrong shape even when labeled as illustrative.

**Principle:** Examples in prompts carry more weight than their explicit labels suggest. Agents reading prompts treat examples as templates for generation, not as labeled illustrations to read past. This means example content has two distinct failure modes: factually wrong examples can model fact-fabrication, and structurally specific examples can over-template the response shape. The mitigation is to choose examples that fail gracefully if mimicked: use patterns (position, era, primary affiliation) over specifics (exact stats, year-precision dates), and verify factual content even in nominally-illustrative examples. Defense-in-depth on examples is cheap; the failure mode of brittle examples is a real cost paid in agent fact-fabrication that's hard to attribute back to the example.

**PM application:** AI products with prompt-engineered behavior frequently include in-prompt examples for tone, format, and structural guidance. The PM-side instinct is to think of examples as illustration — "show, don't tell" — but the engineering reality is that examples in prompts function as templates, not as illustrations. This has implications for prompt review: every example-bearing prompt should be reviewed for "what happens if the agent treats this as a template?" not just for "is the example clear?" Specific failure modes to look for: examples with fabricable specifics (numbers, dates, names) that the agent might mimic into production answers; examples that model behavior counter to other rules in the prompt (the same Griffey-shape conflict from item 4); examples that bake in domain assumptions that drift over time (a 2024-current example becoming stale in 2026). Prompt review checklists should include an "example template-mimic risk" column. The PM who treats examples as load-bearing rather than ornamental ships more reliable AI products.

### Validation against real history surfaces design flaws design alone misses

**Project context:** 2026-05-03, Phase 3B framework v1.5 build, item 11. Reconstructed from `PHASE3B_NOTES.md` item 11 narrative.

**Observation:** The `check_sql_scalar_matches_answer` check was originally drafted to walk the whole trace, looking for the "last qualifying" `run_sql` call — i.e., the last call that returned a single-row, single-column, integer-valued scalar. The design intent was reasonable: skip past intermediate diagnostic queries to find the headline-bearing answer query. When the check was validated against the 8 historical eval runs (80 question-evals total), it produced two false positives on Q6 in two later runs where the agent's last `run_sql` call was a 4-row gametype × HR decomposition. The walk-the-trace logic ignored the multi-row final call and locked onto an earlier intermediate scalar, which the response no longer treated as the headline. The redesign — "last `run_sql` call (and only that one) must qualify on its own; otherwise skip" — both fixed the false positives and produced cleaner semantics: the question the check actually answers became "does the headline scalar match the final query's scalar?" not "does the headline scalar match some scalar in the trace?" The validation work surfaced this within an hour; the original design would have shipped two false-positive noise sources into every future Q6 eval.

**Principle:** Designs that operate over historical or production-distribution data have edge cases that thinking through the design alone reliably misses. Validating the design against actual historical data — not curated test cases, but the real distribution of past observations — surfaces those edge cases concretely. The cost of validation is bounded and small (write the validator, run it across the historical set, inspect the verdicts); the cost of shipping a flawed check is paid in ongoing false-positive noise that erodes trust in the framework. False positives are particularly expensive because each one demands manual triage — the team has to decide "is this a real issue or framework noise?" — and cumulative false positives push teams toward ignoring the framework entirely.

**PM application:** AI products with rule-based or check-based components — content moderation, eval frameworks, automated guardrails, anomaly detection — frequently get tested against synthetic test cases rather than against the messy distribution of real production data. The PM discipline is to require validation against actual historical data before promotion: not the engineer's hand-picked test cases, but a representative sample of what users have actually done. This catches the long tail of "we didn't think anyone would do that" cases that synthetic testing misses by construction. In an iteration-heavy AI product, where rules and checks evolve frequently, building the historical-validation harness as a reusable artifact (per Phase 3B's `replay_historical_evals.py`) pays compounding dividends — every future check change benefits from the same retrospective coverage. The PM who insists on "show me how this performs against the last 100 real cases, not 10 hand-picked ones" prevents a class of post-launch surprise.

### Spec-source choice when validating retrospectively

**Project context:** 2026-05-03, Phase 3B framework v1.5 build, item 12. Reconstructed from `PHASE3B_NOTES.md` item 12 design lessons.

**Observation:** The first version of the replay validation script loaded the historical run's frozen specs from `raw.json` — the spec that was active at the time the run was executed. The intent was reasonable: load everything from the same captured artifact. The behavioral consequence was that all the new `must_not_contain` entries in `eval/benchmarks.py` (Q4's "Colorado vs. Colorado", Q6's "16 home runs in World Series", etc.) were invisible to the validation: historical specs had `must_not_contain=[]`, so the historical hallucinations would not be caught by retrospective application of the historical specs. The fix was to load the current specs from `eval/benchmarks.py` keyed by question id, treating the historical `raw.json` as a source of response text and trace only. This changed the validation question from "how would historical specs evaluate today's responses?" to "how would today's specs evaluate historical responses?" — which is the question the framework iteration actually wants answered. The first formulation would tell us whether old specs were internally consistent against old responses (they were, by construction); the second tells us whether new specs catch the failures we've now encoded against. Substantively different validation questions; surface-similar code paths.

**Principle:** Retrospective validation requires explicit choice about which version of "the rules" applies. Historical data carries its own historical evaluation rules, and applying them gives one answer; applying current rules to historical data gives a different answer. Both questions are legitimate but they answer different things. The default should be: apply current rules — that's what tells you whether your iteration captures the failures you've encoded against, which is the iteration loop's actual purpose. Frozen historical rules answer a question about historical consistency, not about iteration progress. When the spec-source choice isn't deliberate, it's easy to default to whatever the artifact format makes convenient (in this case, frozen specs in `raw.json`), which silently selects the wrong validation question.

**PM application:** When evaluating product changes against historical data — A/B test reanalysis, policy backtesting, retrospective compliance audits — be explicit about which version of "the rules" you're applying. A common trap in AI product work: a new content policy is backtested against historical incidents, but the labels marking "incident" come from old policy applied at the time. The validation now answers "does the new policy align with old policy's labels?" rather than "does the new policy catch the failures we now consider failures?" — the former is internal-consistency checking, the latter is iteration progress. The PM should require the spec-source choice be stated explicitly in any retrospective evaluation: "we are applying current policy rules to historical raw events" vs. "we are validating that historical rule application was internally consistent." Conflating the two makes evaluation results hard to interpret and easy to misuse.

### Skip beats false positive in evaluation infrastructure

**Project context:** 2026-05-03, Phase 3B framework v1.5 build, items 11, 12, 13 considered together. Reconstructed from `eval/checks.py` design notes in `PHASE3B_NOTES.md`.

**Observation:** The three checks built during framework v1.5 (`check_sql_scalar_matches_answer`, `check_threshold_surfaced`, plus the per-question `must_not_contain` enforcement) all share a deliberate SKIP-conservative design. `check_sql_scalar_matches_answer` skips on multi-row, multi-column, non-integer, no-bolded-number-in-response, and no-SQL-in-trace cases — it only fires when it can confidently identify both the headline scalar and the SQL scalar. `check_threshold_surfaced` skips unless the question's `expected_behavior` is `"surface_threshold_and_list_leaders"` — a per-question opt-in that confines the check to questions where threshold prose is genuinely required. The replay harness conservatively SKIPs Q6 in runs where the trace's last query is multi-row, rather than picking an earlier scalar that might or might not match the headline. The choice is explicit: every check pays the cost of additional skip-logic to avoid producing FAIL verdicts on cases the check can't confidently assess. The alternative — fire aggressively, accept some false positives — was rejected at every check.

**Principle:** Evaluation infrastructure that conservatively SKIPs when scope is unclear produces a more trustworthy signal than infrastructure that fires aggressively and accepts false positives. False positives have an asymmetric cost: each one demands manual triage (real issue or framework noise?), and cumulative false positives push teams toward ignoring the framework entirely. False negatives — issues the check could have caught but didn't — have a much smaller marginal cost in a multi-check framework, because other checks (or manual review) will catch the same issues. The SKIP-conservative discipline preserves signal integrity. When a check fires, the team can trust it without re-litigating scope. When a check skips, no one is misled. The cost is a slightly more verbose check implementation (per-call scope-checking logic) and a slightly lower fire rate; both are cheap purchases for trustworthy signal.

**PM application:** AI eval and QA infrastructure should default to SKIP rather than FAIL when the check can't confidently assess the case. Teams that ignore the framework because half of its FAILs turn out to be noise are worse off than teams with a smaller, trustworthy framework. The PM discipline is to push back against "but this might catch something" reasoning when adding broad-scope checks: every check should answer "what are the specific cases where this fires correctly, and what are the edge cases where it should skip?" before being added. The bias should be toward narrow-scope, high-precision checks that compose into framework breadth, not broad-scope checks that accept false-positive noise. This applies beyond eval frameworks: content moderation, anomaly detection, fraud screening, and any system where automated verdicts compete for human triage time. Signal integrity is the whole asset; false positives are an existential threat to it, more dangerous than missed coverage.

### Abstract when the second use case makes variation concrete

**Project context:** 2026-05-03, Phase 3B framework v1.5 build, item 12. Reconstructed from `PHASE3B_NOTES.md` item 12 ("Replay harness generalized... old scratch/validate_check_item11.py deleted").

**Observation:** Item 11's validation work produced a single-purpose script — `scratch/validate_check_item11.py` — that hard-coded the check name, the expected-fail list, and the replay logic. When item 12 needed the same shape of retrospective validation against a different check (the `must_not_contain` entries), the temptation paths were three: copy the script and rename it, build the abstraction in advance during item 11, or generalize at point-of-second-use. The actual choice was the third: item 12's work generalized the script into `scratch/replay_historical_evals.py` with `--check` and `--expected-fail` flags, validated parity against item 11's results, then deleted `validate_check_item11.py`. The abstraction was built when the second concrete case made the variation visible: check name became a parameter, expected-fail list became a parameter, the framework logic stayed shared. Building the abstraction at item 11 time would have required guessing which dimensions were variable; building it at item 12 time used grounded knowledge of the actual variation. Deleting the single-purpose script after parity-verification prevented drift between the specific and the general implementations.

**Principle:** Premature abstraction is more expensive than reactive abstraction in two ways. First, premature abstraction guesses at variability dimensions, frequently wrong — the API ends up wrong-shaped for the second use case and has to be reworked anyway, but now with the cost of an existing abstraction's interface to migrate. Second, premature abstraction builds for use cases that may never arrive; the cost is paid up front for value that may never be realized. Reactive abstraction — wait until the second concrete case arrives, then generalize using the visible variation — pays its cost only when value is confirmed and uses real knowledge of the variation rather than speculation. Two related but distinct disciplines apply here: Rule of Three is threshold-based (count occurrences before abstracting, waiting for three uses to confirm the pattern is real), while the second-use rule is visibility-based (abstract when the variation dimensions become concrete, which can happen on use two). Both are evidence-based, but the evidence differs — frequency versus clarity. The second-use rule applied to item 12 because the variation was already concrete and the cost of duplication was high.

**PM application:** AI product features are particularly susceptible to premature platform thinking — "we should build this as a reusable framework" before there's a second concrete use case. The discipline is to ship the specific thing, then refactor when the second specific thing arrives. This avoids both speculative generality (frameworks built for use cases that never come, which then constrain the actual use cases that do come) and copy-paste duplication (multiple specific implementations that drift apart over time). The PM should resist platformization pressure on first-use features: ship narrow, ship specific, accept the apparent inelegance, refactor at second-use with grounded knowledge of the variation. When the refactor happens, deletion of the original specific implementation is part of the work — leaving both around guarantees drift. The pattern transfers cleanly to ML pipeline tooling, eval frameworks, prompt-engineering harnesses, and any iteration-heavy AI product surface.

### Decision documentation is an engineering artifact

**Project context:** 2026-05-03, Phase 3B framework v1.5, item 14 (recoverable SQL errors and framework signal). Reconstructed from `PHASE3B_NOTES.md` item 14 narrative.

**Observation:** Item 14 was resolved without a code change. The framework already operated in strict mode on `trace_no_errors` — any `run_sql` error in a trace flagged the question as fail, regardless of whether the agent self-corrected. The "resolution" was documenting the design choice with structured reasoning: why strict mode, what alternatives were considered (lenient mode, two-axis tracking with separate `recovery_rate`), why each alternative was rejected, what trade-off was being accepted ("automated fail, behavioral win" cases requiring manual review), and what future signal would warrant revisiting (>20% of fails turning out to be clean recoveries on manual review). The structure mirrors the standard Architecture Decision Record format: context, decision, rejected alternatives, trade-offs, revisit triggers. The artifact's value is independent of any code change — it captures reasoning that would otherwise live only in the head of the engineer who made the call, prevents the question from being re-litigated each time it surfaces, and gives future-self a concrete signal for when to revisit.

**Principle:** Decision documentation is an engineering artifact in its own right, even when no code is written. The default failure mode without it: the same architectural question gets re-litigated each time it surfaces, with progressively less context about why the original choice was made; eventually someone changes the design without realizing what was previously rejected, and the original failure modes the rejection prevented quietly resurface. Documented decisions short-circuit this loop — the next person who asks "should this be lenient mode?" finds the rejection reasoning preserved with full context. The structure that makes this work is consistent across decision shapes: state the decision, surface the alternatives considered, name what each alternative would have traded off, capture the explicit trigger for revisiting. The discipline is light; the value compounds across the project lifetime.

**PM application:** AI product work is decision-dense — model choice, prompt structure, rule scoping, eval framework configuration, fallback policy, latency-vs-quality trade-offs. Many of these decisions never produce code artifacts; they shape what code does without leaving a code trace. The PM application is to require Architecture Decision Records (or equivalent lightweight artifacts) for non-obvious calls, even when the "implementation" is "do nothing different." Without this, the institutional memory of why-we-chose-X gets lost as team members rotate, and the next iteration cycle re-litigates settled questions. ADRs are particularly load-bearing in AI product work because the rapid iteration cadence makes "we already decided this" easy to forget — what was settled six weeks ago is ancient history in an actively-iterating system. The PM who normalizes ADRs as a deliverable, not just code or a Jira ticket, prevents a class of recurring re-debate that wastes team time and risks regression to previously-rejected designs.

### Architectural choices follow project purpose, not absolute "best"

**Project context:** 2026-05-03, Phase 3B framework v1.5, item 14 strict-vs-lenient mode framing. Reconstructed from `PHASE3B_NOTES.md` item 14 reasoning section.

**Observation:** The strict-vs-lenient mode question on `trace_no_errors` looks like an architectural debate with a "right" answer — should the framework treat recovered errors as fails or passes? Item 14's reasoning resolved the question by surfacing the load-bearing premise: "The framework's purpose is to signal correctness for prompt iteration, not to assess whether the agent ultimately produced an acceptable user-facing response." Given that purpose, strict mode is correct: a recovered error reveals a prompt-tightening opportunity (the failed first query indicates the agent's planning context had a gap the prompt should close), and treating it as a pass would hide that signal. But the choice would flip if the framework's purpose were user-facing quality assessment — there, recovery is success, and strict mode would over-fire on traces that produced acceptable user experience. The same code, different purpose, different architecture. The mode choice isn't intrinsically better in one direction; it follows from what the framework is for.

**Principle:** Architectural choices that look like absolute right-vs-wrong debates frequently aren't. They follow from project purpose, and the purpose is often left implicit. When debate stalls between two reasonable architectural alternatives, the productive move is usually upstream: surface the project's purpose explicitly and check which alternative serves it better. If the purpose itself is unclear or contested, the architecture debate is downstream of the more important conversation about goals. The same applies to revisits: when a previously-resolved architectural question re-opens, the most informative test is "has the project's purpose changed?" — if so, the original choice may genuinely warrant reconsideration; if not, the original reasoning still applies and the revisit is unproductive. Purpose is the load-bearing input; architecture is downstream.

**PM application:** AI product architectural debates frequently get stuck at the surface: should we use this model or that model, this guardrail policy or that one, this latency budget or that one. The PM application is to redirect the debate upstream: state the product's purpose explicitly, then check which alternative serves it. Often the "right" choice is obvious once the purpose is stated, and the debate dissolves. When it doesn't dissolve, that's a signal that the team disagrees about purpose — and that's the conversation worth having, not the surface architecture debate. PMs who frequently encounter "but Engineer A says X and Engineer B says Y" stalls should suspect that the project's purpose hasn't been articulated clearly enough to discriminate between the alternatives. Stating purpose is light work with high leverage; it converts unbounded architectural debates into bounded engineering decisions with clear evaluation criteria.

### Naming choices in documentation accumulate meaning

**Project context:** 2026-05-03, Phase 3B closure session. Reconstructed from CLAUDE.md Phase 3B progress narrative and PHASE3B_NOTES status header.

**Observation:** Phase 3B's iteration work was named with three deliberately distinct conventions (as of 2026-05-03), each signaling a different work shape. "Phase 3B Steps 1-4" named the prompt-iteration steps that resolved items via system prompt edits — sequential, additive, sharing a common shape. "Framework v1.5" named the eval-framework engineering work that resolved items 11/12/13/14 — different shape (Python code in `eval/checks.py` and `eval/benchmarks.py`, not prompt edits), different cadence (one batch rather than four steps), different verification surface (replay harness against historical runs rather than fresh eval runs). "Phase 3B closure" named the final session's work on items 3 and 4 — different again (rule reconciliation and stale-evidence closure, neither shaped like prompt iteration nor framework engineering). Naming the closure work "Step 5" would have implied parallel structure to Steps 1-4 that wasn't accurate; readers would expect another item-resolution-via-prompt-edit pattern. The chosen names compress work-shape information that would otherwise need to be re-derived from chronology — future-readers can infer "Steps 1-4 are prompt iteration" without reading the entries, "framework v1.5 is code work" without reading the entries, "closure is end-of-phase reconciliation" without reading the entries.

**Principle:** Naming choices in project documentation are load-bearing in a way that's easy to underweight in the moment. Default naming — sequential numbering, generic version labels, terse identifiers — compresses chronology but loses work-shape information; readers have to re-derive shape from content every time. Deliberate naming chooses words that signal shape, distinguishes structurally-different work even when it lands close in time, and makes the shape of past work legible to future-readers without re-reading the artifacts. The cost of deliberate naming is small (a few minutes of choice per major work increment); the value compounds because every future reader benefits. The discipline applies particularly to multi-step or multi-phase work where the natural temptation is sequential numbering: ask whether the work being named shares shape with the work that came before, and if not, choose a name that signals the difference.

**PM application:** AI product work involves a lot of decision-and-artifact creation that gets named in flight: experiment names, model versions, prompt iterations, eval batches, policy revisions. The PM-side instinct is to default to generic naming (model-v3, prompt-iter-7, exp-Q2-2026-04) because it's fast and uncontroversial. The opportunity cost is real: future team members trying to reconstruct project history have to re-derive what each artifact was actually for. Deliberate naming — names that signal work shape, scope, and distinct character — costs little and pays back across the project's lifetime. The PM application is to budget naming as a real activity, not a clerical one: when introducing a major work increment, choose a name that signals what makes this work distinct from adjacent work, even at the cost of breaking sequential conventions. Names accumulate meaning when they're chosen deliberately; they accumulate noise when they're not.

### Stale backlog items need re-verification before working

**Project context:** 2026-05-03, Phase 3B closure session, item 3 closed-to-watch-list. Reconstructed from `PHASE3B_NOTES.md` items 3 and 18.

**Observation:** Item 3 (era-disambiguation team rule partially ignored) had been carried in the Phase 3B backlog since Phase 3A Step 3 manual testing on 2026-04-30 — three days old by the time the closure session reached it. The closure session's diagnostic surfaced that the evidence had aged past usability: no verbatim response artifact had been preserved, no eval-framework reproduction was possible (no Washington-1920s question existed in `eval/benchmarks.py`), the system prompt had changed substantially across Phase 3B Steps 1-4 plus framework v1.5 since the original observation, and no rule conflict analogous to item 4 could be identified. The "fix" became "close to watch list, no edit" — re-verification revealed that the system being fixed today was meaningfully different from the system the original observation was made against, and a three-day-old single observation against a now-substantially-changed prompt wasn't sufficient evidence to drive a prompt edit. The watch-list closure preserved the option to reopen if the pattern recurs, while not paying iteration cost on stale evidence.

**Principle:** Backlog items lose evidentiary weight over time, and the rate is faster in iteration-heavy systems where the underlying code or prompt is changing rapidly. The system you're fixing today is not necessarily the system the bug was reported against; in an actively-iterating AI system, three days of prompt evolution can be enough to make a previous observation no longer reproduce. Re-verification — not just re-reading the bug report, but checking that the issue still reproduces against current state — should precede engineering work on any non-fresh backlog item. The cheap version is "does the current system, on a question of the same shape, still exhibit the original behavior?" If yes, the diagnosis-and-fix loop is justified. If no, close to watch list with a named reopen trigger, and don't pay iteration cost on a non-issue. This complements the "Diagnosis before fix" discipline: re-verification can short-circuit diagnosis entirely when the issue no longer reproduces, saving the diagnostic work for items that are still real.

**PM application:** AI product backlogs accumulate items at iteration speed and lose freshness at the same speed. The PM trap is treating older backlog items as equally valid candidates for engineering time as fresh items — they're not, because the system they were reported against may have been substantially modified by intervening work. The discipline that pays off is requiring re-verification before scoping engineering work on any backlog item more than a few iteration cycles old: "before we work on this, can we reproduce it against current state?" Items that no longer reproduce should close-to-watch-list with reopen triggers, not be carried indefinitely in active backlogs. This compresses the prioritization conversation and prevents engineering from doing real work on issues that are no longer issues. PMs working in iteration-heavy AI products should treat backlog freshness as a metric in its own right — old items that haven't been re-verified are claims, not facts, and should be priced accordingly.

### Carve-outs in rules over-apply at the edges

**Project context:** 2026-05-02, Phase 3B Step 1, items 1/8/9 ("Trusting the data" section). Reconstructed from `PHASE3B_NOTES.md` items 1, 8, 9 resolution narratives and item 15 (Q4 ballpark-names-from-training) WATCH entry.

**Observation:** Items 1, 8, and 9 were resolved by adding the "## Trusting the data" section to the system prompt, with the load-bearing rule "every numeric claim and game-result detail in your response must come from a query you ran THIS turn." The rule included a carve-out: "names, broad framing, era context are fine from world knowledge; specific numbers, scores, dates, and game-by-game narratives are not." The carve-out had a clear motivation — fully banning all training-knowledge use would block legitimate narrative framing ("during the steroid era," "before the DH was introduced"). But the carve-out's "broad framing" language proved interpretable. At item 15's Q4 verification, the agent included ballpark names ("Comiskey Park," "Coors Field") in the response without querying `retro.ballparks` — sourced from training knowledge, factually correct in this run, but routed through the carve-out's "broad framing" interpretation rather than through the queried data the rule was supposed to compel. The carve-out for legitimate narrative framing over-applied to a category (ballpark names) that arguably should have been queried; the agent's interpretation was internally consistent with the rule's letter but defeated its spirit.

**Principle:** Carve-outs in rules — exceptions, qualifiers, "except when" clauses — over-apply at the edges. The rule's intent is communicated through the strict portion; the carve-out is communicated through the lenient portion; readers (human or agent) frequently interpret the carve-out generously and the strict portion narrowly, because that's the easier interpretive path. The mitigation is twofold: scope carve-outs as narrowly as possible (specific named categories rather than broad descriptors), and pair every carve-out with explicit non-coverage ("but this does NOT include X, Y, Z"). Both increase prompt length, but the cost is bounded; the cost of an over-applied carve-out is paid in failure modes that the rule was supposed to prevent quietly resurfacing through the exception. When the rule matters, the carve-out should be tighter than feels necessary — readers will widen it back to "feels necessary" through interpretation.

**PM application:** AI products with rule-based behavior — content policies, response guidelines, disclaimer requirements, citation rules — frequently include carve-outs to handle edge cases without over-restricting normal behavior. The PM application is to treat carve-outs as load-bearing surface area, not as parenthetical refinements: every carve-out should be reviewed for what it might cover that the rule's intent wouldn't. Specific failure shapes to look for: vague descriptors ("broad framing," "common knowledge," "non-controversial") that admit broad interpretation; carve-outs that name categories without naming what's excluded from the carve-out; carve-outs added retroactively to handle a single observed false-positive without considering what other cases the carve-out now covers. Carve-out review checklists should include "what's the worst-case interpretation of this exception?" — and if the worst case violates the rule's intent, tighten the carve-out before shipping. Rule effectiveness is determined as much by what gets exempted as by what gets prohibited.

### Positive-behavior instructions need stronger brakes than negative-prevention

**Project context:** 2026-05-02 through 2026-05-03, Phase 3B Steps 2-3 and closure session, items 7 and 4. Reconstructed from `PHASE3B_NOTES.md` items 7 (Dynamism section landing) and 4 (rule reconciliation closure).

**Observation:** The Phase 3B prompt iteration produced two distinct shapes of rule. Items 1, 8, 9 were resolved with negative-prevention rules — "every numeric claim must come from a query you ran THIS turn." The rule landed cleanly, didn't drift across iterations, and verified consistently across multiple eval runs. Item 7 was resolved with positive-behavior rules — the "## Dynamism — multi-interpretation when warranted" section instructing the agent to answer multiple interpretations when questions are genuinely ambiguous. The Dynamism rule was correct against its motivating observation but had broader behavioral surface area than negative-prevention rules: there are many ways to "answer multiple interpretations," and the rule didn't bound which way. Phase 3B closure (item 4) discovered that the Dynamism rule had silently extended into territory governed by the disambiguation logic rule — narrative + multi-match questions where the disambiguation rule said `ask_user` and the Dynamism rule said "answer all interpretations." Both rules were individually correct; the positive-behavior surface area of Dynamism overlapped with the constraint surface of disambiguation, and neither rule had cross-references to brake the extension.

**Principle:** Positive-behavior instructions ("do X when Y") have more degrees of freedom than negative-prevention instructions ("don't do Z"). Negative-prevention has a single concrete shape — the prohibited behavior is well-defined and the agent avoids it. Positive-behavior has a range — many ways to satisfy the instruction, and the agent picks one of them, which may not be the one intended. That range of acceptable interpretations is where conflicts with adjacent rules can hide silently. This is closely related to the carve-out drift pattern: both involve readers interpreting permissive rule surface generously. Carve-outs widen exception scope; positive-behavior rules widen acceptable-behavior scope. Same drift direction, different rule structures. The mitigation is to apply stronger brakes when authoring positive-behavior rules: bound the range of the behavior shape explicitly (count limits, format constraints, scope qualifiers), pair the rule with cross-references to adjacent constraints, and choose examples that constrain the shape rather than expand it. Negative-prevention rules generally don't need this — the prohibited behavior is its own brake. Positive-behavior rules without brakes drift to fill available behavioral surface, and the surface they fill may include territory other rules were supposed to constrain.

**PM application:** AI product prompt engineering frequently distinguishes "do this" rules from "don't do this" rules but treats them as symmetrically authored — both get a sentence or paragraph in the system prompt, both get reviewed against the motivating observation, both get verified by eval. The asymmetry that matters is in surface area: positive-behavior rules cover a range of acceptable behaviors; negative-prevention rules cover a single prohibited behavior. The PM application is to require additional review on positive-behavior rules: what's the maximum reasonable interpretation of this instruction? Does it overlap with any existing rules' territory? Are the bounds explicit, or are they implied by the example? When in doubt, prefer narrower scoping — agents widen instructions through interpretation, they don't tighten them. The pattern transfers cleanly to non-LLM AI product surfaces as well: any rule-based system where the rule allows for multiple satisfying behaviors is at risk of drifting into adjacent constrained territory.

### Prompt size has real cost paid in latency

**Project context:** 2026-05-03, Phase 3B Step 4 (item 6, the 23-quirks consolidation). Reconstructed from `PHASE3B_NOTES.md` item 6 acknowledged-cost section.

**Observation:** Phase 3B Step 4 added a "### Schema gotchas — consult before writing SQL on these columns" section to the system prompt, addressing 13 column-level data quirks plus 3 cross-cutting notes. The section was the right work — Phase 3A Step 2 only included 5 quirks, and the missing 23 were documented in CLAUDE.md §4 with no prompt-side coverage. The eval runs verified the new section produced behavioral wins (item 13 took on Q3, item 8 self-resolved on Q6) and no regressions on existing questions. The cost surfaced in the same eval: latency rose from 313s baseline to 522s post-item-6 — a 67% increase on the 10-question eval, attributable to prompt size growth from ~150 to ~280 lines. Most questions were 1.5-3× slower; only Q1 and Q7 stayed flat (likely cache warming). The cost was acknowledged and accepted at the time — the prompt was now feature-complete on quirks, and future iterations could be additive without further bloat — but the data point matters: doubling the prompt size produced a 1.67× latency increase, against zero observed quality regression. Future expansion would compound this cost. The Phase 3D notes registered a future option of moving MED/LOW-priority gotchas to a `lookup_column_notes(column)` tool, lazily consulted only when relevant columns appear in the planned SQL.

**Principle:** Prompts feel weightless when authored — each new instruction is reviewed against its motivating observation, not against the cumulative cost of all instructions. The cost is real but invisible at the per-instruction level: it's paid in latency at every invocation, in attention budget the model allocates at every read, and in maintenance overhead when rules drift or conflict. The cost surfaces only when measured at the prompt-as-a-whole level — and by that point, the prompt is large and individual instructions are hard to remove without re-litigating the items they resolved. The mitigation is to budget prompt size as a constraint, not a free resource, and to evaluate alternative architectures when the prompt approaches the budget: lazy consultation via tool calls (per the Phase 3D `lookup_column_notes` option), conditional inclusion via dynamic prompt assembly, or factoring rarely-applicable rules out to a separate retrieval step. Each architecture trades latency or complexity against prompt size; the right choice depends on which dimension the project is constrained on.

**PM application:** AI products that iterate via prompt engineering accumulate prompt size at iteration speed and rarely shrink it. The PM application is to track prompt size as a quality metric — token count, line count, or section count — and to set a budget that triggers architectural review when exceeded. The trigger doesn't dictate the response (sometimes prompt growth is the right answer; sometimes a tool-based factoring is better); it forces the conversation to happen rather than letting prompt size drift past the point where alternatives could have been considered cheaply. Latency-cost-per-prompt-line is a measurable quantity in any LLM product (run an eval at two prompt sizes, divide); making the cost visible to engineering and product stakeholders makes "should we add this rule?" a real question rather than a default-yes. PMs working in iteration-heavy AI products should treat prompt size review as a recurring activity, not a one-time exercise — without explicit checkpoints, prompt-size-driven latency creeps into user experience without anyone owning the trend.

### Evaluation frameworks grow specific through use

**Project context:** 2026-05-03, Phase 3B framework v1.5, item 12 (per-question must_not_contain entries). Reconstructed from `PHASE3B_NOTES.md` item 12 narrative ("framework grows specificity through use").

**Observation:** The v1 eval framework launched 2026-05-02 with general checks: does the answer match `expected_answer`? Does the response contain required keywords (`must_contain`)? Are there no errors in the trace? These checks caught broad categories of failure — wrong-number answers, missing topic coverage, unrecovered SQL errors. But manual review of the v1 baseline run surfaced specific hallucinations the general checks missed: Q4 wrote "Colorado vs Colorado" for Bichette and Perez; Q6 inferred "16 home runs in World Series play" from arithmetic rather than data; Q8 referred to "previous record of 71" for a Bonds prior season that didn't exist. None of these failed the general checks — they were factually wrong in ways that didn't trip the existing assertion shapes. Item 12 encoded the specific observed hallucinations as `must_not_contain` entries on Q4, Q6, and Q8 — explicit substring patterns that fail the question if they appear in any future response. The framework gained per-question specificity that the v1 design didn't include. Defensive variants ("Colorado vs Colorado", "vs. Colorado (home)", "vs Colorado (home)") were added prophylactically — they cost nothing to encode and cover punctuation/word-order variations of the same hallucination. The framework's specificity is a memory of "things we've seen go wrong," encoded in code.

**Principle:** Evaluation frameworks grow specific through use, not at the design phase. The v1 framework starts with general checks because that's what's known — the failure modes haven't been observed yet. As the system iterates and produces specific failure modes, the framework accumulates specific guard rails against those exact failures. Generic checks become joined by specific checks; total framework size grows monotonically because past failures don't un-occur. This is distinct from per-check skip-discipline (separate entry): that pattern is about how an individual check should handle uncertainty when designed; this pattern is about how the framework as a whole should evolve over time. Per-check skip is about scope at the check level; framework specificity growth is about coverage at the framework level. This pattern is asymmetric with the system being evaluated: the system might shrink (refactor, simplify), but the eval framework against it should not — every removed check is a forgotten failure mode that may recur. The discipline that pays off is to expect specificity growth, plan for it (per-question spec architecture, easy substring-pattern additions, defensive variants), and not treat the eval as a fixed artifact. Frameworks that don't grow specific over time are either operating on systems too simple to surface specific failures, or are missing the iteration loop that should be feeding them.

**PM application:** AI product eval/QA infrastructure should be designed to grow specific through use, not stay clean and general. Teams that resist specificity growth ("our eval framework is generic, that's elegant") miss the failure modes that drive product quality. The PM application is to budget for ongoing eval specificity work: each surfaced failure mode should produce a corresponding eval entry, defensive variants of common phrasings should be added prophylactically, and the eval framework should be treated as institutional memory of past failures rather than as a clean test suite. The cost is bounded (each entry is small), the value compounds (every future regression test runs every entry), and the alternative — re-discovering past failures in production — is much more expensive. PMs working in iteration-heavy AI products should normalize specificity growth as a sign of healthy framework maturation, and resist requests to "clean up" the eval suite by removing specific checks: those specifics are the framework's value, not its noise.

### Skills as architectural fork — and when not to take it

**Project context:** 2026-05-03 morning session. Mid-project, Robert raised a question after observing Phase 3B Step 4 had increased latency 67% from prompt size growth. The exchange surfaced an architectural fork that had been implicit in the project's monolithic-prompt design from Phase 3A onward.

**Observation:** Robert asked whether the project could use "skills" — focused, named capability modules loaded on demand — instead of accumulating everything in a single growing system prompt. The question came from genuine curiosity rather than from a specific pain point, which made it the right moment to surface the architectural conversation rather than dismiss it. The senior-advisor response had two parts. First: explain skills concretely (small folders containing demand-loaded instructions/knowledge, each with a name and description so the agent decides when to load), describe how they'd map to Baseball Oracle (core identity always loaded; SQL writing always loaded; schema gotchas, era awareness, and disambiguation logic loaded only when relevant), and surface real trade-offs (engineering complexity, decision overhead, testing complexity for combinations, debugging complexity). Second: recommend not doing it now, because the project was producing a working agent with measurable improvements, the current monolithic prompt was iterating cleanly, skills require new infrastructure (loading mechanism, registry, possibly a different SDK pattern), and the latency cost from item 6 hadn't yet reached "acutely painful" — it was early signal of a real scaling problem, not the problem itself. The recommendation was to track latency as a metric going forward and revisit skills if/when the pain became sharp.

**Principle:** Architectural forks worth surfacing aren't always architectural forks worth taking. The discipline that pays off is making the fork visible — explaining what the alternative architecture is, what problem it solves, what costs it introduces, and what conditions would make it the right choice — even when the recommendation is to stay on the current path. This separates "I don't know what alternatives exist" from "I know the alternatives and am deliberately staying with the current design." The first is ignorance; the second is informed pragmatism. Many architectural mistakes come from teams that didn't know the alternatives existed when they made the original choice; visible-but-deferred alternatives prevent that failure mode without committing to premature refactoring. The trigger for revisiting should be specific (e.g., "if latency becomes user-visibly painful" or "if we want to add another large knowledge section"), not vague.

**PM application:** AI products that iterate via prompt engineering reach scaling problems eventually — prompt size growth, rule-conflict accumulation, latency cost, attention-budget exhaustion. The standard architectural answer is some form of demand-loaded capability modules (called "skills" in some frameworks, "tools-as-knowledge" in others, "conditional context loading" when hand-rolled). The PM application is to surface this fork to the team early in the project's life — not to commit to it, but to make sure the team knows it exists before the prompt grows past the point where refactoring is cheap. Track prompt size and latency as quality metrics that trigger architectural review when thresholds are exceeded. The trigger doesn't dictate the response; it forces the conversation. PMs who normalize "we know about X, we're not doing it now, here's the trigger that would change that" prevent both premature optimization (refactoring before the cost is real) and accumulated technical debt (refactoring when it's expensive because the cost was always implicit). The visible-deferred-alternative pattern transfers to model selection, infrastructure choices, and any architecture decision that has an obvious-but-not-yet-justified alternative.

The pattern transfers beyond AI: any time the immediate work is also a vehicle for capability building, stating that layering changes what the work should look like and what feedback it should produce. PMs who treat collaboration framing as a real input — alongside scope, timeline, and resources — get better outcomes on both layers.

### Three-shape engineering work in one day

**Project context:** 2026-05-03, Phase 3B closure session combined with framework v1.5 work. Reconstructed from `PHASE3B_NOTES.md` status header chronology.

**Observation:** The May 3 session ran three structurally distinct shapes of engineering work consecutively: prompt iteration (item 4 disambiguation rule reconciliation, item 3 close-to-watch-list — both produced edits to the system prompt at `agent/prompts.py`), framework engineering (items 11, 12, 13 framework v1.5 — produced Python code changes in `eval/checks.py` and `eval/benchmarks.py` plus a new replay harness), and design decision documentation (item 14 ADR — produced no code change, only structured prose capturing the strict-vs-lenient choice). Each shape had different review surface (prompt-text-level vs. code-diff-level vs. prose-level), different verification cadence (eval re-run vs. retrospective replay vs. no automated verification), and different "done" criteria (verified at fresh eval vs. parity-checked against historical runs vs. reasoning captured with future-revisit trigger). The recognition that these were distinct shapes — not just three items in a backlog — emerged when the closure-session naming convention split into "Phase 3B closure" (prompt iteration), "framework v1.5" (code engineering), and unnumbered ADR work (design decision). Treating them with the same review lens would have either over-reviewed the ADR work (no code to diff) or under-reviewed the framework work (substantial new code).

**Principle:** Engineering work has structurally distinct shapes that benefit from being recognized as such, even when they land close in time. Prompt iteration optimizes a system prompt against observed behaviors; framework engineering builds or modifies code that operates on the system; design decision documentation captures reasoning without changing any artifact except the document itself. Each shape demands different review tools, different verification surfaces, and different signals of completion. Code review reads diffs; prose review reads for argumentative clarity and missing alternatives; eval review reads pass/fail patterns for regression and behavioral fit. Different attention, different artifacts read. Treating them as interchangeable backlog items collapses the distinctions and produces either misallocated review attention or misnamed completion criteria. The discipline that pays off is identifying the shape of work explicitly when it's planned, choosing review and verification approaches that match the shape, and naming the work in a way that signals the shape to future-readers (the naming-accumulates-meaning pattern). Sessions that contain multiple shapes can be productive — the shapes don't conflict — but they require deliberate transitions between review modes, not a single uniform approach across the session.

**PM application:** AI product work routinely contains shape mixing — prompt edits, code changes, and design decisions all land in the same iteration cycle. The PM trap is treating them as a uniform backlog where each item gets the same kind of review and the same kind of "done" definition. The discipline that pays off is recognizing the shape of each item, allocating review attention accordingly (code review for code changes, prose review for ADRs, eval-run review for prompt iterations), and ensuring the team has the right specialists for each shape — code review needs an engineer, prompt iteration may need product input on UX expectations, ADR review needs someone who understands the architectural context. PMs who treat all AI-iteration work as "engineering" lose the differentiation that makes review effective. Shape-aware planning also helps with pacing: a session that contains three shapes is denser than a session that contains three prompt edits, and recognizing the difference helps with scope-setting and time estimation. The pattern transfers beyond AI — any engineering work where multiple deliverable shapes are common (specs, code, runbooks, dashboards) benefits from shape-aware review structure.

### Behavioral wins can produce automated noise — and that's signal, not bug

**Project context:** 2026-05-03, Phase 3B framework v1.5, item 14 (recoverable SQL errors and framework signal). Reconstructed from `PHASE3B_NOTES.md` item 14 acknowledged-trade-off section and verification at `eval/results/2026-05-02_084228/`.

**Observation:** The eval framework operates `trace_no_errors` in strict mode — any `run_sql` error in a trace flags the question as fail, regardless of whether the agent self-corrected on retry. This produces a class of traces that the framework labels FAIL but that produced fully correct user-facing responses. Q4 in `eval/results/2026-05-02_084228/` is the canonical example: the agent's first SQL hit `opp.nick` as a column-name truncation from the SELECT alias `opp_nick`, the agent self-corrected in one retry, and the final response was fully correct (correct cycles, correct opponents, correct teams). The automated framework: FAIL. The user-facing behavior: WIN. Item 14 documented the asymmetry as deliberate — the failed first query reveals a prompt-tightening opportunity (the agent's planning context had a gap that the prompt should close), and treating it as a pass would hide that signal. Manual review distinguishes "automated fail, behavioral win" (clean recovery, prompt opportunity) from "automated fail, behavioral fail" (broken response or unrecovered error). Both are real fails from the framework's perspective; only one is a user-visible problem. The cost is interpretation overhead on every flagged trace; the benefit is signal integrity for the prompt iteration loop.

**Principle:** Automated metrics and behavioral quality are not always aligned, and a high-signal automated framework will produce a class of cases where the metric fires on something the user wouldn't have noticed. Suppressing the metric to "match user experience" is the wrong move — the metric is catching a real signal that's actionable for iteration (the failure conditions exist in the system, even if recovery hides them this run). Conflating automated signals with user-experience signals is also wrong — they answer different questions. This is downstream of the "Architectural choices follow project purpose" pattern (separate entry): that pattern is about how to choose between strict and lenient when designing the framework; this pattern is about how to operate the framework after strict mode is chosen, recognizing that strict-mode signals will diverge from user-experience signals and treating the divergence as informative rather than noisy. The discipline is to keep both kinds of signal visible, distinguish them in review, and use each for its appropriate purpose: automated metrics for iteration leverage, user-experience signals for product quality assessment. The asymmetry is informative, not noise. Frameworks that produce zero "automated fail, behavioral win" cases are likely operating in lenient mode and missing iteration opportunities; frameworks that produce many such cases without manual reconciliation are likely losing trust because every signal demands triage.

**PM application:** AI products that iterate via automated evaluation will produce signal asymmetries between automated metrics and behavioral outcomes. The PM trap is choosing between strict mode ("fail on any deviation") and lenient mode ("fail only on user-visible deviation") as if it's a single choice with a right answer. The discipline that pays off is keeping both lenses available: the iteration loop runs against strict-mode signals (which catch every deviation, including recoverable ones) while the product quality loop runs against behavioral assessment (which catches user-visible failures). The two loops answer different questions and shouldn't be collapsed. Manual review reconciles them — when a strict-mode automated FAIL turns out to be a behavioral WIN, that's prompt-tightening territory, not framework noise to suppress. PMs who treat "automated metric matches behavior" as a goal to achieve are making the wrong target visible; the right target is "automated metric catches every deviation, behavioral assessment catches every user-visible failure, and the gap between them is the iteration opportunity space." The pattern applies to any AI product where automated assertions and user-facing behavior can diverge — content moderation, agent reliability, generation quality assessment.
